# GPU 服务器调度课程设计：项目框架、研发流程与三人分工（讨论稿）

> 当前状态：只用于讨论总体方案，暂不创建 GitHub 仓库、分支或程序文件。  
> 技术语言建议：C++17。  
> 核心目标：在保证所有输出绝对合法的前提下，通过复现论文与工业调度器中的有效思想，持续提高公开数据集和测试赛上的调度质量。

## 1. 我们真正要解决的事情

这不是一个“写出能运行的调度程序”就结束的普通课程设计，而是一个带排行榜的组合优化问题。

最终成绩同时受三个指标影响：

1. 加权等待时间：高优先级任务等待越久，惩罚越大；
2. GPU 显存空闲：服务器在整个调度周期内的显存浪费越多，惩罚越大；
3. 最晚完成时间：全部任务结束得越晚，惩罚越大。

任何约束被违反，该实例的成绩都会遭受严重损失。因此项目目标分为两个层级：

- 底线目标：对任何输入都能在 60 秒、1GB 内存、单 CPU 核心环境中稳定地产生合法解；
- 竞争目标：尽可能逼近优秀方案，在测试赛中通过排名反馈持续优化。

这两个目标不能互相替代。一个复杂但偶尔非法的算法不如一个稳定的普通算法；一个永远合法但缺乏优化能力的算法也难以获得好排名。

## 2. 项目的完整组成

整个项目不应只有一个 `scheduler.cpp`，而应由下面八个部分组成。

### 2.1 问题模型与输入输出层

职责：

- 读取服务器和任务数据；
- 定义 `Server`、`Task`、`Assignment`、`Schedule` 等数据结构；
- 计算任务在不同服务器上的最小 GPU 分配数；
- 输出严格符合评测要求的结果；
- 确保标准输出中没有日志、提示语或调试信息。

特别需要预计算：

- 每个任务可以运行在哪些服务器上；
- 任务在每台可行服务器上至少需要多少张 GPU；
- 每个任务的可行服务器数量，即“适配稀缺度”；
- 每台服务器对不同任务的资源匹配程度。

### 2.2 调度状态与快速查询层

这是算法的基础设施，负责回答：

- 某个任务最早什么时候能在某台服务器上启动；
- 某个时间区间内 GPU、CPU 和内存是否足够；
- 插入、删除或移动任务后，资源状态如何快速更新；
- 当前有哪些任务已经提交但尚未调度；
- 下一次任务提交或任务完成事件是什么时候。

这一层的性能会直接决定高级搜索算法能尝试多少个候选解。

建议优先研究两种表示：

1. 事件驱动状态：只在提交、开始、完成等关键时间点更新；
2. 时间区间结构：按服务器维护资源占用区间，支持快速查找可行空隙。

不宜在时间跨度很大时简单建立“服务器 × 每个时间片”的巨大数组，否则可能浪费内存和搜索时间。

### 2.3 合法性验证器

验证器独立检查：

- 是否每个任务恰好出现一次；
- 开始时间是否不早于提交时间；
- 完成时间是否等于开始时间加运行时长；
- GPU 分配是否满足数量和显存需求；
- 服务器是否满足单任务的 GPU、CPU、内存要求；
- 同一时刻服务器上的并发任务是否超出任何资源；
- 任务是否始终在同一服务器连续运行。

验证器是所有优化工作的安全网。每个算法候选版本、每个本地测试输出都必须经过它。

### 2.4 本地指标与实验评估器

本地评估器需要计算三个原始指标：

- `E_wait`：加权等待时间；
- `E_memory`：平均 GPU 显存空闲；
- `E_finish`：全部任务最晚完成时间。

官方最终分数使用全班合法方案的最小值和最大值进行归一化，因此本地无法精确复现排行榜分数。我们的本地工具应：

- 保存三个原始指标，而不是粗暴地只合成一个分数；
- 与当前基线和历史最佳版本逐实例比较；
- 分简单、中等、困难、极端四档统计；
- 标记“某指标变好但另一指标显著变差”的情况；
- 在获得测试赛结果后，反推更合理的内部评价权重。

### 2.5 基线算法层

至少保留多个可随时运行的基线：

1. 官方示例算法；
2. 保证合法的简单贪心；
3. 优先级优先贪心；
4. 最早可启动优先贪心；
5. 最佳适配服务器贪心；
6. 当前团队最稳定的综合贪心。

基线不是“写完就扔掉”的临时代码。所有高级策略都必须通过消融实验说明它比哪些基线更好、好在哪里。

### 2.6 核心优化算法层

这是项目的竞争核心，建议分成三个逐步增强的层次。

#### 第一层：调度规则与组合贪心

为任务构造优先级评分，例如考虑：

- 原始优先级权重；
- 已经等待的时间；
- 任务运行时长；
- 最低 GPU 数；
- GPU 显存、CPU、内存需求；
- 可行服务器数量；
- 是否为资源贴边任务；
- 是否会阻塞其他稀缺任务。

为“任务—服务器—开始位置”构造放置代价，例如考虑：

- 新增等待时间；
- 新增最晚完成时间；
- GPU 显存碎片；
- GPU 数量碎片；
- CPU、内存剩余碎片；
- 是否占用了该任务不需要的高价值服务器；
- 是否破坏后续稀缺任务的可行空间。

这一阶段重点不是找一个神奇公式，而是建立可配置、可比较的评分框架。

#### 第二层：多起点与随机化构造

同一套贪心评分不只运行一次，而是在 60 秒预算内：

- 使用不同参数组合；
- 对排名靠前的候选任务做受限随机选择；
- 使用不同任务排序规则；
- 使用不同服务器选择规则；
- 保留当前最优合法解。

这种方式实现成本低、稳定性高，通常是课程竞赛中性价比很高的一步。

#### 第三层：局部搜索与大邻域搜索

从一个合法解出发，反复破坏并修复：

- 将单个任务移动到另一台服务器或另一开始时间；
- 交换两个任务的服务器或顺序；
- 选出高等待任务重新插入；
- 选出造成严重资源碎片的任务重新插入；
- 选出关键路径末端的长任务重新安排；
- 清空某一服务器的局部时间窗口后重新调度；
- 对一组相互冲突的稀缺任务联合重排。

可借鉴的方法包括：

- Local Search；
- Iterated Local Search；
- Large Neighborhood Search；
- Simulated Annealing；
- Tabu Search；
- GRASP；
- Beam Search。

不需要一次实现全部。应通过论文调研和小规模实验，优先选择最容易适配当前约束、单位开发时间收益最高的方法。

### 2.7 论文与工业方案研究层

论文和工业代码不是单独写一篇读书笔记，而是要进入研发流水线。

优先调研方向：

1. HPC / GPU cluster scheduling；
2. heterogeneous cluster scheduling；
3. online and offline job scheduling；
4. multi-resource scheduling；
5. bin packing with time dimension；
6. RCPSP（资源受限项目调度）；
7. parallel machine scheduling；
8. Kubernetes scheduler 的过滤与打分机制；
9. Slurm 的 backfilling、priority 和资源选择机制；
10. Apache YARN / Mesos 的多资源公平调度思想；
11. Borg / Omega 等集群调度系统；
12. Tiresias、Gandiva、Pollux 等面向深度学习任务的调度研究；
13. OR-Tools CP-SAT、局部搜索和 LNS 的建模思想。

每项研究必须形成一张“策略卡片”：

| 字段 | 内容 |
|---|---|
| 来源 | 论文、项目或技术文档 |
| 原问题 | 它解决什么问题 |
| 核心机制 | 算法最重要的思想 |
| 与本题差异 | 是否允许抢占、迁移、弹性 GPU 等 |
| 可移植部分 | 哪个策略能用于当前题目 |
| 实现成本 | 低 / 中 / 高 |
| 预期收益 | 主要改善哪个指标 |
| 实验结果 | 对哪些实例有效 |
| 最终决定 | 采用、修改采用或放弃 |

研究的目标不是完整复刻一个工业系统，而是抽取其中适合本题的部件。例如：

- 从 Kubernetes 借鉴“Filter + Score”两阶段候选选择；
- 从 Slurm 借鉴保留高优先级任务启动机会的 backfilling；
- 从多资源调度借鉴 dominant resource 和资源平衡思想；
- 从 LNS 借鉴“破坏一部分解，再用强贪心修复”；
- 从 bin packing 借鉴 best-fit decreasing 和碎片控制；
- 从 RCPSP 借鉴串行调度生成方案和优先规则。

### 2.8 工程、提交与报告层

职责包括：

- CMake 或直接 g++ 构建；
- `build.sh` 和 `run.sh`；
- Linux、C++17 环境兼容；
- 运行时间和内存控制；
- 批量运行 100 个公开实例；
- 自动检查非法输出；
- 保存每次实验的参数、Git commit 和结果；
- 打包成规定的一层目录结构；
- 为英文报告和答辩保留算法演化、测试提升与分工证据。

## 3. 整个项目的研发流程

项目应按下面的闭环运行，而不是三个人分别埋头写三个孤立模块。

```text
阅读题目与分析数据
        ↓
阅读论文 / 工业实现
        ↓
提出一个可验证的算法假设
        ↓
在统一框架中实现策略
        ↓
合法性验证
        ↓
100 个公开实例批量实验
        ↓
与基线和历史最佳逐实例比较
        ↓
分析在哪类实例有效或失效
        ↓
保留、修改或删除策略
        ↓
参加测试赛并利用隐藏集反馈
        └──────────────→ 返回研究与改进
```

每次算法改进都应回答五个问题：

1. 改动来自什么观察、论文或工业策略？
2. 它预计改善哪个指标？
3. 它在哪些类型的实例上应该有效？
4. 实际结果是否符合预期？
5. 增益是否值得增加的复杂度和运行时间？

## 5. 分工原则：三人都做算法，各自再守住一个支柱

不建议采用下面这种分法：

- 一个人只写算法；
- 一个人只测试；
- 一个人只写报告。

原因是算法性能占据项目核心，而且论文理解、算法实现和实验分析不能完全分离。更合理的是：

- 三个人都阅读论文或工业方案；
- 三个人都至少实现一个可比较的调度策略或搜索算子；
- 每个人另外承担一个长期主责，保证项目基础设施有人负责到底；
- 任何重要算法至少有两个人理解，避免形成单点故障。

## 6. 建议的三人角色

先用 A、B、C 代称，之后再按三个人的兴趣和能力交换。

### 成员 A：构造算法与调度状态主责

长期主责：

- 核心数据模型；
- 服务器资源状态；
- 最早可行开始时间查询；
- 基线和综合贪心框架；
- 算法主流程的运行效率。

算法研究方向：

- list scheduling；
- best-fit / first-fit decreasing；
- 稀缺任务优先；
- 多资源放置评分；
- Kubernetes 的 Filter + Score；
- Slurm priority/backfilling 中可适配的思想。

必须交付：

- 至少两种构造型基线；
- 可插拔的任务排序和服务器评分接口；
- 一份构造策略对比实验；
- 与 B 共同实现至少一个高级搜索修复器。

### 成员 B：搜索优化与验证主责

长期主责：

- 独立合法性验证器；
- 指标计算器；
- 解的插入、删除、移动和回滚操作；
- 时间预算控制；
- 局部搜索和大邻域搜索框架。

算法研究方向：

- GRASP；
- Iterated Local Search；
- Large Neighborhood Search；
- Simulated Annealing；
- Beam Search；
- OR-Tools 中的局部搜索与 LNS 思路。

必须交付：

- 独立验证器和评分器；
- 至少三种邻域操作；
- 一个“破坏—修复”搜索流程；
- 与 A 共同优化搜索过程中最耗时的数据结构。

### 成员 C：实验科学与策略研究主责

长期主责：

- 100 个公开实例的数据画像；
- 批量运行、结果保存和版本比较；
- 参数实验和消融实验；
- 论文、开源项目和工业方案的策略卡片；
- 将实验结论沉淀为报告和答辩素材。

算法研究方向：

- 多起点随机化贪心；
- 自动参数组合选择；
- 按实例特征选择算法；
- multi-resource fairness / dominant resource；
- GPU 集群调度论文中的等待与碎片优化；
- 针对困难、极端实例的专用排序规则。

必须交付：

- 数据集特征分析；
- 可重复的批量实验工具；
- 至少一个多起点或自适应策略；
- 每轮迭代的实验结论；
- 至少实现一个实际进入主程序的算法策略，而不是只整理资料。

## 7. 三人的共同工作

以下工作不属于某一个人的“杂活”，而是全员职责：

1. 每人负责调研若干论文或工业调度器；
2. 每人提出至少两个可实验的算法假设；
3. 每人至少实现一个调度策略或邻域算子；
4. 每人参与代码审查；
5. 每人都要能解释最终主算法；
6. 每轮测试赛后三人一起做结果复盘；
7. 报告中的个人贡献以 Git 提交、策略卡片和实验记录为依据。

## 8. 第一轮研究任务如何拆分

为了避免三个人读到重复内容，第一轮可以按三个视角并行：

### A 线：工业集群调度

重点：

- Kubernetes Scheduler；
- Slurm priority、backfill 和资源选择；
- Borg / Omega；
- YARN / Mesos。

目标：

- 找到任务过滤、服务器打分、优先级提升、回填和碎片控制的可迁移思想。

### B 线：组合优化方法

重点：

- RCPSP；
- parallel machine scheduling；
- multidimensional bin packing；
- GRASP、LNS、局部搜索和模拟退火。

目标：

- 设计适合本题的解表示、邻域结构、破坏算子和修复算子。

### C 线：GPU / 深度学习集群论文

重点：

- Tiresias；
- Gandiva；
- Pollux；
- Themis；
- 其他异构 GPU 集群和多资源调度工作。

目标：

- 找到优先级、等待时间、GPU 碎片和异构服务器匹配方面的有效启发式。

第一次讨论时，每人不需要做长篇汇报，只需提交 3～5 张高质量策略卡片，并明确建议先实现哪一个思想。

## 9. 版本演化计划

### V0：合法闭环

目标：

- C++17 可构建运行；
- 所有公开实例都能输出合法解；
- 能计算三个本地指标；
- 能批量运行和比较结果。

这一版本不追求排名，但没有它，后续优化无法可靠进行。

### V1：多规则贪心

目标：

- 多种任务排序；
- 多种服务器评分；
- 稀缺任务保护；
- 资源碎片惩罚；
- 多起点选择最优结果。

这是第一次测试赛最现实的主力版本。

### V2：局部搜索

目标：

- move、swap、reinsert；
- 优先修复高等待任务；
- 优先处理关键路径末端任务；
- 在固定时间预算中持续改进。

### V3：大邻域和自适应

目标：

- 多种破坏算子；
- 强贪心修复；
- 根据实例规模和资源紧张程度选择策略；
- 动态分配 60 秒运行预算。

### V4：隐藏集导向优化

目标：

- 根据测试赛反馈定位公开集过拟合；
- 调整内部目标权重；
- 为困难和极端实例设计稳健策略；
- 删除收益不稳定但复杂度很高的模块。

### V5：正式赛冻结

目标：

- 不再引入高风险大改；
- 全量回归测试；
- 超时、内存、非法输出和打包检查；
- 固定随机种子或设计稳定的种子策略；
- `main` 分支只保留最终稳定版本。

## 12. 算法策略的优先级

按照预期性价比排序：

### 第一优先级

- 独立验证器和批量实验工具；
- 多种贪心规则；
- 服务器 Filter + Score；
- 稀缺任务优先；
- 资源碎片代价；
- 多起点随机化；
- 保留当前最佳解。

### 第二优先级

- move / swap / reinsert 局部搜索；
- backfilling 思想；
- LNS 破坏与修复；
- 根据实例特征切换策略；
- 参数组合搜索。

### 第三优先级

- Beam Search；
- 模拟退火或禁忌搜索；
- 更复杂的区间数据结构；
- 小规模子问题精确求解。

### 暂不优先

- 完整复刻 Kubernetes、Slurm 或某篇大型系统；
- 直接建立覆盖整个时间轴的大型整数规划；
- 依赖评测环境未提供的外部求解器；
- 需要多线程或 GPU 才能发挥作用的方案；
- 不适用于“不可抢占、不可迁移、运行时长固定”的论文机制。

## 15. 当前推荐结论

项目的核心不应被理解成“一个人写主算法，另外两个人辅助”，而应是：

> 三个人共同组成算法研究组；A 负责高效构造，B 负责搜索优化，C 负责实验驱动与自适应策略；三人分别守住核心状态、正确性、实验系统三个工程支柱。

这样的分工既能把主要人力投入性能竞争，又不会因为缺少验证、实验或工程稳定性而让算法成果无法落地。
