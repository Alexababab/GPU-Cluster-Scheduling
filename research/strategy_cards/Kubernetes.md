# Filter + Score 调度框架策略卡
* **策略名称**：Kubernetes Scheduling Framework (Filter + Score Mechanism)
* **来源**：Z. Rejiba and J. Chamanara, "Custom Scheduling in Kubernetes: A Survey on Common Problems and Solution Approaches", ACM Comput. Surv., 2022.
* **原问题**：
* 容器化工作负载在集群中的最优节点分配问题。
* 调度需满足各类硬约束（Hard constraints）与软约束（Soft constraints）。
* 需适应多变的应用需求（如机器学习、边缘计算网络延迟等）。
* **核心机制**：
* **两阶段调度流程**：首先进行“过滤”（Filtering），排除不满足硬约束（如资源不足）的节点，生成可行节点子集。 其次进行“打分”（Scoring），基于一组软约束对可行节点进行加权评分。 最后将容器组绑定到得分最高的节点。
* **插件化扩展框架（Scheduling Framework）**：允许在调度流程（如排序 Sorting、过滤 Filtering、打分 Scoring）中插入自定义逻辑（Plugins）。 此框架将调度器转变为可插拔组件，便于根据工作负载需求动态启用或禁用特定逻辑。
* **多维度队列管理**：通过修改 Sorting 行为，可支持批处理调度（Batch Scheduling）与多策略作业排队。
* **与本题的差异**：
* Kubernetes 默认调度器主要面向长期运行的在线服务和通用容器。 本题处理的是固定运行时长、不可抢占、不可迁移的离线 GPU 任务。
* Kubernetes 是基于事件触发的在线调度系统。 本题是拥有全局视野的静态离线调度优化问题，需一次性输出完整调度结果。
* **本题可采用部分**：
* **Filter + Score 架构**：可作为构建本题 V1 阶段多规则综合贪心算法的基础框架。
* **Sorting 扩展点**：参考 Kubernetes 自定义队列排序逻辑，实现针对本题的多种任务排序规则（如提交时间优先、高权重短任务优先）。
* **Scoring 的加权组合设计**：参考原生加权机制，将不同维度的启发式目标（如减少内存碎片、保护稀缺资源）线性叠加以得出最优服务器分配
* **计划修改**：
* **Sorting（任务排序）阶段**：将原生任务队列排序替换为针对离线任务的静态与动态排序框架，支持按优先级权重、可行服务器数量、多 GPU 需求等特征进行切换与组合。
* **Filtering（服务器过滤）阶段**：除了常规检查，必须过滤掉无法在候选时间区间内放置任务的服务器，并严格执行 GPU 数量、显存、CPU、内存四维硬约束的检查。
* **Scoring（服务器评分）阶段**：设计完全适用于本题的软性评分特征，如最早开始时间代价、GPU 数量碎片、显存与系统资源剩余碎片、服务器稀缺价值。
* **预计改善指标**：
* $E_{finish}$（最晚完成时间）：通过合理的综合评分平衡异构集群负载，防止长时任务堆积。
* $E_{wait}$（加权等待时间）：通过 Sorting 阶段高权重任务优先策略，减少高优任务的等待延迟。
* $E_{memory}$（显存空闲）：通过多维度碎片惩罚评分（Scoring），引导任务最佳匹配，减少单机资源隔离与浪费。
* **预计适用实例**：
* 适用于所有难度档位的公开测试集实例。
* 特别适用于需要多种启发式规则结合以跳出单一贪心陷阱的中等、困难及极端规模实例。
* **实现成本**：低-中。 在 C++17 中基于面向对象多态或函数指针，易于实现高内聚可配置的 Task Sorting 与 Filter + Score 接口。

* **实验对照**：
* V0 基准：采用最佳适配（best-fit）的事件驱动简单贪心算法。
* 消融实验：单独测试针对单一的 Sorting 策略或单项 Score 特征的正交对比表现。
* 综合测试：在时间预算内结合多起点随机化（Multi-start），对比多种规则组合的综合评分上限性能。
